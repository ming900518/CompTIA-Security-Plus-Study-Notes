### Security+ 中定義的術語

以下的術語解釋皆出自參考書翻譯而成

> 我覺得我翻的跟機翻差不多，如果你有其他解釋，那就用你自己的會比較好理解。

| 術語                                                | 解釋                                                                                                               |
| --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| Acceptable Use Policy / Rules of Behavior           | 用公司或組織所訂定的準則，來規範員工及成員該如何使用公司或組織的資源。例如電腦及網路存取。                         |
| Annual Loss Expectancy (ALE)                        | 一年內某個風險會造成的損失。                                                                                       |
| Annualized Rate of Occurrence (ARO)                 | 一年當中某個風險的發生次數。                                                                                       |
| Asset Value (AV)                                    | 跟現金流有關的財產價值。                                                                                           |
| Business Impact Analysis (BIA)                      | 分析當業務所需的資源遭受干擾時，會對業務產生的衝擊。                                                               |
| Business Partners Agreement (BPA)                   | 用來劃清業務活動中參予的成員所承擔的責任、義務和損益分配的一種同意書。                                             |
| Exposure Factor (EF)                                | 當發現威脅時，潛在的資產損失百分比。                                                                               |
| Interconnection Security Agreement (ISA)            | 資訊系統服務提供方與操作方同意與其他組織互聯的技術要求文件。本文件通常會提供在組織間的備忘錄中。                   |
| Maximum Tolerable Downtime (MTD)                    | 業務最大可被中斷，不會影響組織生存的時間。                                                                         |
| Mean Time Between Failures (MTBF)                   | 某系統或某元件預計能夠存活的時間。                                                                                 |
| Mean Time to Failures (MTTF)                        | 某系統或某元件會失敗的平均時間。                                                                                   |
| Mean Time to Restore (MTTR)                         | 某系統或某元件從失敗到恢復所需的平均時間。                                                                         |
| Memorandum of Understanding / Agreement (MOU / MOA) | 即「諒解備忘錄」，用來定義彼此間的權利義務以達成共同追求的目標。                                                   |
| Recovery Point Objective (RPO)                      | 在系統失效前，最後資料正常且可用來還原系統的時間點。                                                               |
| Recovery Time Objective (RTO)                       | 在後果可承受的範圍內，容許服務或進程中斷的最長時間。                                                               |
| Redundant Array of Independent Disks (RAID)         | 即 RAID 磁碟陣列，一種可以用來提供容錯率的配置，有很多種陣列可用。                                                 |
| Risk                                                | 無論意外或意識的前提下，數個威脅會發生並會留下系統風險或損失的可能性。                                             |
| Risk Acceptance                                     | 一種用來應對風險的策略，當風險發生時可用來選擇會留下最少後果的狀況。                                               |
| Risk Analysis                                       | 評估可以被識別的風險，每一個風險應該都要能列出大綱、解釋與發生的可能性。                                           |
| Risk Assessment                                     | 評估風險存在的可能性，在做每一個決策前都應該先做這項評估。                                                         |
| Risk Avoidance                                      | 一種以「避免風險」來處理風險的策略。                                                                               |
| Risk Caculation                                     | 用花費、數量或頻率等方式計算風險數                                                                                 |
| Risk Deterrence                                     | 一種以「使攻擊者無攻擊慾望」來處理風險的策略。                                                                     |
| Risk Mitigation                                     | 一種以「降低風險」來處理風險的策略。                                                                               |
| Risk Transference                                   | 一種以「保險、第三方合約、共同責任」來處理風險的策略。                                                             |
| Service-Level Agreement (SLA)                       | 一紙有規範廠商效能（用 MTBF 和 MTTR 來界定效能）的同意書。                                                         |
| Single Loss Expectancy (SLE)                        | 預計單一損失發生時的花費，損失可以是致命的失敗或遭攻擊後的結果。                                                   |
| Single Point of Failure (SPOF)                      | 一個會使所有系統下線得單一弱點。                                                                                   |
| Vulnerability                                       | 系統於資安層面、設計或實作上的設計問題或弱點，或內部控制中有可能在意外或意識下泄漏並有可能造成違反安全原則的事件。 |

### 威脅判定

資安上的威脅能進行以下的分類

- **環境的威脅：** 天災或意外發生
- **人為的威脅：** 操作不當
- **內部的或外部的威脅：** 內部指的是組織內，例如員工。外部則相反。

### 風險判定

在對資安威脅進行風險判定時，過程中通常會由以下幾個元素組成，必須針對這個情境來建構應對的方式

1. 在作業系統、伺服器軟體和程式中有許多已知威脅，要找到一個應對這些威脅的最佳解決方法。
2. 應該要包含一份能用來判斷風險可能發生的機率分析報告，這樣能讓公司更好地分配資源。
3. 透過與 BIA 結合，可以給組織認知到面臨資安問題的嚴重性，也能讓組織在面對不同的場景時能選擇更加智慧的決定。

### 風險判定的計算

當做風險判定時，最重要的是將風險做重要度排列。同樣的風險在不同的環境下發生會產生不同的後果，這點也應該納入考慮。要做出排列，就必須對風險做計算。  
風險計算的公式如下：

    SLE * ARO = ALE

這個計算可以根據「停止服務時長」或「金錢」等不同角度來解讀，由於上面這個公式會將風險量化為數字，故本公式為金錢導向的。用不同的角度來解讀會導致重要度排序較為不一樣，也會影響組織做決策時的判斷。

我們還應該要對 SLA 中所規範的範圍與硬體的限制有所理解，下面幾點是應該熟悉的指標：

1. **風險發生的可能性：** 通常在各種報告中會用「極高」、「高」、「中等」、「低」、「極低」來做分別，也可以透過量化的數值來判定（10, 8, 5, 2, 0）
2. **攻擊向量（Threat Vectors）：** 這項指的是攻擊者可以用來對付你的工具或他們可以跟隨的攻擊路徑，攻擊向量可以指任何從攻擊工具到系統弱點的任何會被攻擊者所使用的事物。

### 判定隱私

有兩個跟隱私有關的評估是必須被熟悉的：PIA 跟 PTA

**PIA（Privacy Impact Accessment，隱私衝擊評估）**  
通常和 BIA 相關，說明收集的 PII（Personally Identifiable Information，個人可識別資訊），並提出這些資訊是任何被使用、存取、分享、保護及儲存的，這份評估也會提到當這些資訊面對風險時會如何被識別以及如何降低風險

**PTA（Privacy Threshold Assessment，隱私門檻評估）**  
這份報告通常會在 PIA 前製作，會分析使用的系統中有什麼需要保護的隱私資訊，有些單位會以此評估是否製作 PIA

> 資料來源：[https://www.dhs.gov/compliance](https://www.dhs.gov/compliance)，上面有圖，還蠻好理解的

### 根據風險判定行動

當已經識別出風險並分析其可能發生後，下面是可能的下一步動作：

- **避免風險（Risk Avoidance）：** 使會讓風險發生的前提消失。
- **轉移風險（Risk Transference）：** 用其他服務（如保險）來彌補風險發生後的損失。
- **降低風險（Risk Mitigation）：** 利用不同的手段，將風險發生的可能性降低。
- **接受風險（Risk Acceptance）：** 就......我知道風險但我也不能幹嘛，就希望它不要發生吧（啥都不做）。

前三個為主動的動作，需要在風險發生前就被執行。而最後一個是被動的。

### 與雲端計算相關的風險

有幾種實現雲端計算的方法，如下所述：

- **軟體即服務（Software as a Service，SaaS）：** 廠商利用網路提供軟體，並以此軟體作為服務。如 Google Drive 和 iCloud 等等。
- **平臺即服務（Platform as a Service，PaaS）：** 這些廠商提供讓使用者在平臺上建構軟體，並讓軟體在該平臺上執行的服務。如 Google Apps Engine。
- **基建即服務（Infrasture as a Service，IaaS）：** 這些廠商提供硬體或虛擬機，讓用戶可以完整控制該環境上的任何設定。如 Amazon Web Services 中的 EC2 和 Google Cloud Engine 中的 Compute Engine。

在實現上述雲端計算的功能前，下面有一些規則需要注意：

- **管理承諾（Regulatory Compliance）：** 並不是所有的資料與服務都在組織的控制與管理下，在這情況下，需要建立一些承諾，並要求提供資料存儲或服務的組織和自己的組織規範相同。
- **使用者權限（User Privileges）：** 強制使用者的權限可以被合理的負擔，如果使用者沒有獲得「最少的權限」（least privileges），用戶額外獲得的權限，可能使他們有意或無意的取得他們所不該取得的資訊，並使他們曝露在風險中。特別要注意的是，在雲端管理帳號並不像在本地一樣，而且如果有服務使用者或有心人士無心或有意的將帳號鎖定了，你可能會需要在服務提供商有辦法回應的時間內要求他們處理。
- **檔案的整合與分離（Data Integration/Segregation）：** 服務提供商為了最大化其收益，可能會把數間公司的資料存放在一起，為了避免發生問題，應該要將存放在服務提供商伺服器中的資料做加密或改存至其他較為安全的伺服器中。而為了避免資料因為意外而被不該取得權限的使用者取得，資料應該整合在一起管理。

### 與虛擬化相關的風險

虛擬化有以下幾個風險：

- 有意人士可能透過破壞虛擬機或虛擬層的方式，存取到不該被存取的資料。
- 混亂的網路及安全控制，用來管理虛擬機的程式並不是每一個都擁有相同的設定，這可能會造成特權提升（privilege escalation）的問題，進而降低資訊安全的強度。

現在有非常多的虛擬化威脅都集中在 Hypervisor 上，如果 Hypervisor 被突破了，攻擊者就可以取得所有在該 Hypervisior 之上的客戶端的管理員權限，最好的防範方式便是常常更新虛擬機程式及其上的客戶端系統。

> Hypervisor 這個詞似乎沒有正式的中文翻譯 😂 稍微解釋一下，Hypervisor 有兩種：  
> 一種是「託管 Hypervisor」，建立於主要 OS 之上，這種 Hypervisor 需要接受主要 OS 的管理，效能較差，常見的 VMware Workstaton 和 VirtualBox 就是這種。  
> 而另一種是建立在主要 OS 之下的「原生 Hypervisor」，它會將主要 OS 也納入其管理，甚至有些更為極端的做法是直接將 Hypervisor 當做主要 OS 使用，效能較好，使用這個類型的虛擬機有 VMware ESXi（本身就是 OS）、Hyper-V（Windows NT 的內核元件，Hyper-V Server 本質上還是一個 Windows NT based OS 所以跟一般 Windows 並沒有不同） 與 KVM（是 Linux 的一個 Kernel Module）。

### 建立原則、標準與指導方針

在建立及維護一個安全的網路環境時，必須要先有一個原則（Policies）、標準（Standards）與指導方針（Guideline），這可以幫助確定方向、提供權力及給與你要求其他人達成目的的話語權。

原則用來爲一個大問題提供一個最高指導原則，標準告訴什麼事是需要被注意的，而指導方針則被用來提供如何達成目標的建議。

### 實作原則

下面是製作**原則**的一些關鍵：

- **作用範圍：** 這個原則適用的範圍，並提供讀者了解這原則是幹嘛的。

  > Scope 我原本要翻譯成作用域 😂 ~~碼農後遺症~~ 這東西通常言簡意賅，一句話就帶過了 ~~反正也沒人看（誤~~

- **原則大綱：** 大綱提供原則的目標、這原則重要的理由和要如何服從原則。

  > 同樣，不要講幹話，一段帶過就是了

- **原則：** 當讀者了解重要性之後，就把原則的內容**清楚的、不含糊的**在這寫出來，並沒有特定的呈現方式，愛用文章、列表或 checklist 都行。
- **負責：** 這邊會規定原則生效後，哪一個職位的成員有義務去使原則被正確的套用，也告訴讀者當出現問題時要找誰處理 ~~背鍋~~，通常在這裡還會說明如果沒有服從原則的後果。
- **例外：** 就算是最好的原則都沒辦法預知每一種可能性，這邊就用來規範如果脫離原則時的處理方式與步驟，通常會有要向上呈報的要求

**標準**用來處理商業活動中特定的問題或觀點，標準會從原則為出發點開始制定，反正下面有五點製作標準的關鍵：

- **作用範圍跟用途：** 解釋這份標準的用途與動機。如果這個標準是為了一個技術實作而建立的，那作用的範圍就應該要把軟體、更新、附加元件還有相關的資料全部放到裡面去。
- **角色與責任歸屬：** 規定負責實作、監管與維持標準的人，這個規定並不代表其他人不能超過這些規定，而是在出現問題的時候可以找到人負責
- **引用資料：** 解釋這個標準跟組織中的原則之間的相關性，可以讓有疑問的人 ~~刁民~~ 去找標準的源頭，你也可以透過這些引用資料，當他人質疑標準時可以拿來佐證。
- **性能準則：** 說明要達成指標的方式，通常要包含相關的標準線跟科技標準。

  > 簡單來說，就是要把軟硬體規格都寫出來然後給出效能大概在哪個等級醬

- **維護與管理要求：** 說明管理系統或網路會需要知道的東西

  > 不僅要說明維護中會用到的東西，維護的產品、零件什麼時候要更換也需要記載在裡面。

**指導方針**會記載要怎麼實作標準規範，有點類似操作手冊的文件，製作這份文件有四個關鍵：

- **作用範圍跟用途：** 廢話不多說，就跟上面差不多，你還是得告訴讀者這份指導方針是給誰看的，用在什麼用途上。
- **角色與責任歸屬：** 如果讓沒有經過訓練或沒有能力的人員去處理，可能會導致意外的結果，所以跟上面一樣要把角色規定好，然後把責任歸屬說清楚 ~~才能找人背鍋~~
- **指導內容：** 這個部分就得寫的比較詳細了，需要把操作的過程完整的寫在裡面，最好是搞 Step by step 的比較好讀，然後由於是給操作的人看的，可以把內容寫的簡單點或提供建議或小撇步。
- **行動考量：** 有些行動是不需要天天操作的，在這邊可以規範要在什麼時候執行什麼行動。

但除了上面那些，還有十六個是需要被實作的：

- **在商業活動上的原則：**

  - **給廠商的原則：**

    包含標準作業程序（SOP）、服務級別協定（SLA）和商業夥伴協定（Business Partner Agreements, BPAs），這些文件規範了責任義務還有分攤營收與損失，也同時幫助了商業活動運作的正常與穩定。

    除了以上這些文件外，簽屬 MOU 也是很重要的，MOU 規範了使用條款和確認了權責，在 MOU 內也要規範結束前的條款。

    最後一份文件叫做互聯安全協定（Interconnection Security Agreement, ISA），這份文件會規範組織間技術與安全層面的要求以建立、運作與維持資料間的互聯，同時也要包含資安要求以保護系統與資料。

    > 雖然 MOU 裡也會包含 ISA 規範的內容，但 ISA 是偏向技術面的，MOU 應該不包含任何技術內容。

  - **給員工的原則：**

    - **義務假：** 除了給員工休息之外，公司也能藉此測試當某人不在崗位上時對工作的影響，進而改善員工互相替代的能力。
    - **崗位輪換：** 提供員工一個崗位輪換的最長時間，跟義務假差不多，也能用來測試公司是否太依賴某個成員。
    - **分散責任原則：** 這個原則讓員工能互相監督，降低員工個人的權力，降低被詐欺的風險和損失。

      > 舉個例子：如果 A 員工有權限自己調動財產並自行簽核，公司就有了被 A 員工侵佔財產的風險。這時如果將管理權或財產調動權分給 B 員工，A 員工要能成功侵佔公司財產就得買通 B 員工，相對來說較為困難，對公司而言風險就較小。

    - **「乾淨的桌子」：** 不要把文件就擺在桌上，保護起來。
    - **背景調查：** 調查一下新進員工的背景，確保沒有競爭對手的員工混進來。
    - **保密協議：** NDA 都有簽過吧，反正就是不准把機密帶出去。
    - **入職管理：** 讓新員工能快速進入工作狀態，也要讓他們快速了解公司的規定避免發生資安風險。
    - **繼續教育：** 讓老員工能了解最新的技術，CompuTIA Security+要三年更新就是基於這理由 ~~該死~~
    - **離職面談：** 當員工要離職的時候可以聽聽他們對組織的看法。
    - **以角色為原則的訓練：** 不是每個員工都有相同的權限，而根據角色的不同，組織應該要為他們提供専屬的訓練，而訓練的內容也應該要遵循「最少權限原則」以確保沒有人可以為自己提權，讓組織的資訊或財產的風險降低。
    - **可接受使用原則：** 建立這個原則規範員工使用組織資源或在工作崗位上使用個人電子產品的方式，這樣能確保組織財產的安全並避免未經授權的操作帶來的風險。
    - **一般安全性策略：** 這個文件用來規範什麼控制要素需要被套用來維護系統安全

- **網路及程式使用策略：** 公司的財產都會在法庭與市場上代表著公司，所以公司的資訊產品不得用來執行個人事物。

當你發現想要處理的風險後，最好的做法應該是將其納入控制，以下是控制的類別與控制：

偽陰性偽陽性就跟 COVID-19 常講的那個一樣，在資安上會以 **錯誤類別（Error Type）** 表示：第一級為偽陽性，第二級為偽陰性，第三級則是意外發生後才發現其實早有預兆。

### 風險管理的最佳做法

維持業務連續性的最好方法就是做**BIA**，以及將過去在業界中被證實結果最好的做法彙整出來作為**最佳做法**。先從 BIA 開始，營運衝擊分析（Business Impact Analysis, BIA）用來分析與判斷當一個重要的系統遭受衝擊時的狀態，並以上述結論擬定恢復計劃。

下面列出幾個需要在 BIA 中記載的重點：

- **識別關鍵功能：** 組織必須知道在服務錯誤時，有什麼功能是需要維持運作直到服務重新上線，這樣才能更好的建立需要先恢復以在最短時間內重新開始業務的系統清單，每一個部分都需要向上檢核以確保沒有任何系統被遺漏。
- **需要優先考慮的關鍵業務功能：** 業務運作相關系統重新上線後，需要排定其他系統上線，而在這邊需要做重要性考慮，由於組織提供了用於恢復系統的資源可能不是完整的。
- **計算嚴重系統失敗的時間範圍：** 組織能在重要功能無法使用的情況下存活多久是個重要的課題，故組織能在多久時間內恢復運作是重要的課題，當面對衝擊時，需要將以下要素考慮進去：
  - **生命：** 有沒有人的生命安全受到威脅？
  - **財產：** 有沒有任何事物會損失？
  - **安全：** 有沒有任何事物被這次衝擊傷害到？
  - **金融：** 有多少金額會因為這次的業務中斷而損失？
  - **聲望：** 這次的業務損失會導致怎麼樣的信任損失？

以上這些可以用來制定用來當衝擊發生時需要執行的事情以縮小損失。

- **估計組織中實體與非實體的衝擊：** 提前知道組織會有什麼業務、系統及服務在面臨衝擊時中斷，會增加你的組織在面對衝擊時的反應速度。

總之 BIA 能達到以下幾項有組織性的目標：

- 使組織了解當服務中斷時，真正的衝擊和損壞是可視的。
- 使組織了解服務中斷的代價有多大進而幫你爭取經費。
- 最重要的是，在這過程中會記錄之前曾用過的策略，幫助你的組織在類似情況中快速恢復。

前面有提到識別關鍵的系統，其實所有關鍵的系統及元件都需要被考慮進去，比如水電，發生天災時，可能無法透過組織的資源快速恢復那些東西。當在判斷業務連續性時，這些災難兜需要被計算進去。如果可能的話，建立一個沒有單點故障（Single Point Of Failure, SPOF）可能性或與其連結的服務以確保最佳的業務持續性。

作為一個管理員，一定要隨時注意問題可能隨時發生並盡量有個如何解決的想法。當然人是不可能會知道每一個會發生的緊急事故，但可以就能想到的部分準備計劃，以下有幾個方法可以納入考量：

- **自動化與腳本：** 可以嘗試做個 script，當偵測到錯誤時自動執行以在最快時間內恢復。
- **框架與模板：** 如果之前有發生過類似的事件，可以建立成模板，當之後又發生時直接把模板拿出來快速套用即可。
- **主映像：** 有些作業系統提供系統映像，當問題發生時直接把映像倒回去就完事了。
- **非持久性：** 比如你的系統中有做「快照」（Snapshot），當問題發生時把系統救援工具開起來，還原到正常的快照就完事了
- **彈性：** 很多雲端平臺提供將系統資源提高的服務，可以在試圖修復的過程中加速，盡量縮短服務停用的時間。
- **可擴展性：** 服務使用的系統資源太高了怎辦，加錢買設備啊（笑），這時可擴展性就很重要了，可以在短時間內增加資源。
- **分散分佈：** 通常又被稱作負載均衡（load balancing），將服務分散在多個系統中，讓多個系統一起分擔。
- **高可用性（High Availability, HA）：** 在某個甘泉的地方設立完全相同的服務，當主要服務個挖掉的時候可以立刻切換過去，在用戶端那邊完全看不到任何差異，目標是讓關鍵服務能維持在 99.999%的可用性（又稱為五個九可用性）。
- **為恢復力做計劃：** 有些東西在 IT 的世界中是難以被解決的，比如伺服器房炸了，這個時候就要去計算說怎麼做是比較經濟實惠的，並用結論來管理那些風險。

  > 行話「風險恢復力」指的就是風險管理。

- **冗餘：** 這個詞的意思是說系統中有重複或有提供故障轉移（failover）的部分，現代的伺服器作業系統都有以叢集的方式提供故障轉移的功能，出問題切過去就完事了。
- **錯誤容忍：** 這指的是當系統中有個元件失效時還能正常運作。備用零件和備援電力是絕對不能忽視的，通常會用 N+1 的方式，亦即擺一個備品在旁邊以便立刻替換。而電力的部分，通常 Server 都會有至少兩顆 PSU（電源供應器）以避免 PSU 炸了直接斷電，如果電力源頭出了問題可以透過備援發電機供電，而等待發電機啓動的這段時間則可以透過 UPS 來提供短時間的電力需求。
- **容錯式磁碟陣列（RAID）：** 透過硬體轉接卡或軟體連結的方式將多顆硬碟組合在一起，根據不同的配置有不同的代號，如果同時套用多種配置就把代號兜寫出來，比如做了 RAID 0 跟 RAID 1 就會寫成 RAID 10，以此類推。

  > RAID 2、RAID 3、RAID 4 我就不提了，很少人用，RAID 6 很少考，基本上就是 RAID 5 的增強版。然後硬體的陣列比軟體快。

  - **RAID 0（又稱為 Striped）：** 最少需要兩顆硬碟才能組成，容量是兩顆硬碟的容量相加，將資料拆一半，然後將資料分散寫在兩顆硬碟中。

    > 意思就是一顆硬碟爆了你就只剩另外 50%的資料，而那 50%的資料是根本無法閱讀的，這個磁碟陣列通常被拿來搞高速存儲而非用來做資料安全相關的事。

  - **RAID 1（又稱為 Mirrored）：** 最少需要兩顆硬碟才能組成，容量只有兩顆硬碟中其中一顆，將資料存到兩顆硬碟去，等於是直接幫你做了一份備份。

    > 感覺好像很好，但它的缺點是你買了 1 單位的容量但只能使用到 0.5 單位，非常的不經濟實惠。

  - **RAID 5：** 最少需要三顆硬碟，容量是 (硬碟數-1)\*容量，將一顆硬碟的容量用來儲存奇偶校驗值，並分散儲存在每顆硬碟中，這樣當其中一顆硬碟掛了還能透過計算奇偶校驗值的方式把資料還原出來。

    > 比 RAID 1 還經濟實惠，但就是犧牲了安全性。

### 變更管理

最大的風險常出在當組織面對變更時，變更管理的規則注重在如何記錄與控制那些變更，這也是專案管理的一個子項目。變更包括但不限於實體硬體、個人跟組織本身，在記錄變更時應該要把所有的變更以及其影響的事物記錄下來。
